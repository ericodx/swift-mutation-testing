# Stryker JSON Report Compatibility

The `--output` flag writes a JSON file intended to be compatible with the [Stryker mutation testing report schema](https://github.com/stryker-mutator/mutation-testing-elements/tree/master/packages/report-schema) (version 1). This document records what is compatible, what deviates, and the practical impact of each deviation.

## Schema reference

The Stryker report schema (v1) is defined and maintained at:

```
https://github.com/stryker-mutator/mutation-testing-elements/tree/master/packages/report-schema
```

The same schema is consumed by:

- **mutation-testing-elements** ‚Äî the web components used to render the interactive HTML dashboard (also used by other Stryker language implementations)
- **Stryker Dashboard** ‚Äî the hosted report viewer at `dashboard.stryker-mutator.io`
- Any third-party tool that accepts Stryker-formatted reports (e.g., GitLab mutation coverage widgets)

The report is generated by `JsonReporter` and written to the path specified by `--output`.

## Top-level fields

| Field | Stryker schema | Our output | Compatible |
|---|---|---|---|
| `schemaVersion` | `string` | `"1"` | ‚úì |
| `thresholds.high` | `integer` | `80` | ‚úì |
| `thresholds.low` | `integer` | `60` | ‚úì |
| `projectRoot` | `string?` (optional) | absolute project path | ‚úì |
| `files` | `{ [path: string]: FileResult }` | relative paths as keys | ‚úì |

The `thresholds` values are hardcoded at `high: 80`, `low: 60`. These are the Stryker defaults and cannot currently be configured via CLI or YAML.

## File-level fields

Each entry in `files` corresponds to one source file that contains at least one mutant.

| Field | Stryker schema | Our output | Compatible |
|---|---|---|---|
| `language` | `string` | `"swift"` | ‚úì |
| `source` | `string` | full UTF-8 file content | ‚úì |
| `mutants` | `MutantResult[]` | see below | partial |

File paths in the `files` dictionary are relative to `projectRoot` (the absolute project path is stripped from the front of each file path).

## Mutant-level fields

Each object in the `mutants` array describes one mutation point and its outcome.

| Field | Stryker schema | Our output | Compatible |
|---|---|---|---|
| `id` | `string` | `"swift-mutation-testing_<n>"` | ‚úì |
| `mutatorName` | `string` | operator identifier (e.g., `"RelationalOperatorReplacement"`) | ‚úì |
| `originalText` | `string?` | original token(s) (e.g., `">="`) | ‚úì |
| `replacement` | `string?` | mutated token(s) (e.g., `">"`) | ‚úì |
| `location.start.line` | `integer >= 1` | 1-based line number | ‚úì |
| `location.start.column` | `integer >= 0` | 1-based column number | ‚úó ‚Äî see [deviation 1](#deviation-1-column-numbers-are-1-based) |
| `location.end.line` | `integer >= 1` | same as start line | ‚úì |
| `location.end.column` | `integer >= 0` | `start.column + originalText.count` | ‚úó ‚Äî see [deviation 1](#deviation-1-column-numbers-are-1-based) |
| `status` | enum (see below) | string | partial ‚Äî see [status mapping](#status-mapping) |
| `killedBy` | `string[]?` | `string?` (single value or null) | ‚úó ‚Äî see [deviation 2](#deviation-2-killedby-is-a-string-not-an-array) |
| `description` | `string?` | `"<original> ‚Üí <mutated>"` | ‚úì |

## Status mapping

The Stryker schema defines a fixed set of valid status values. The table below maps each of our internal statuses to the closest Stryker equivalent and records whether the emitted string matches exactly.

| Our `ExecutionStatus` | Emitted string | Stryker schema value | Compatible |
|---|---|---|---|
| `killed(by:)` | `"Killed"` | `"Killed"` | ‚úì |
| `killedByCrash` | `"Crash"` | `"RuntimeError"` | ‚úó ‚Äî see [deviation 3](#deviation-3-crash-and-unviable-are-not-stryker-status-values) |
| `survived` | `"Survived"` | `"Survived"` | ‚úì |
| `timeout` | `"Timeout"` | `"Timeout"` | ‚úì |
| `noCoverage` | `"NoCoverage"` | `"NoCoverage"` | ‚úì |
| `unviable` | `"Unviable"` | `"CompileError"` | ‚úó ‚Äî see [deviation 3](#deviation-3-crash-and-unviable-are-not-stryker-status-values) |

## Known deviations

### Deviation 1 ‚Äî Column numbers are 1-based

**What we emit:** `location.start.column` and `location.end.column` are 1-based, following SwiftSyntax's `SourceLocation` convention. For a token at the very beginning of a line, we emit `column: 1`.

**What Stryker expects:** column values `>= 0`, i.e., 0-based. A token at the beginning of a line should be `column: 0`.

**Impact:** mutant location highlighting in `mutation-testing-elements` and the Stryker Dashboard will appear shifted one character to the right relative to the actual mutation site. The report is still parsed and rendered ‚Äî it is purely a display shift.

---

### Deviation 2 ‚Äî `killedBy` is a string, not an array

**What we emit:**

```json
"killedBy": "testShouldReturnFalseWhenAgeIsBelow18"
```

Or `null` when the mutant was not killed by name.

**What Stryker expects:**

```json
"killedBy": ["testShouldReturnFalseWhenAgeIsBelow18"]
```

The schema defines `killedBy` as `string[]` ‚Äî an array even when there is a single killer.

**Impact:** strict schema validators will reject the field. The `mutation-testing-elements` web component may silently ignore `killedBy` or fail to render the killing test name in the mutant detail panel. The rest of the report (score, file breakdown, status) is unaffected.

---

### Deviation 3 ‚Äî `Crash` and `Unviable` are not Stryker status values

**What we emit:** `"Crash"` for mutants that killed the process via a runtime crash, and `"Unviable"` for mutants that did not compile.

**What Stryker expects:** `"RuntimeError"` is the closest equivalent to `"Crash"`. `"CompileError"` is the closest equivalent to `"Unviable"`. Neither `"Crash"` nor `"Unviable"` appear in the Stryker v1 schema.

**Impact:** mutants with these statuses are treated as unknown by `mutation-testing-elements` and the Stryker Dashboard. They may be omitted from the rendered report or displayed without a recognised status label. They do not affect the score shown in the report, because the Stryker schema computes score from `Killed` and `Survived` counts ‚Äî unrecognised statuses are ignored in that calculation.

In practice:
- `"Crash"` mutants are killed (the mutation was detected). Their contribution to the kill count is lost in Stryker-rendered views.
- `"Unviable"` mutants should not count toward the score at all ‚Äî consistent with how our own score excludes them ‚Äî so their omission from Stryker's rendering is acceptable.

## Summary

| Concern | Status |
|---|---|
| Report can be parsed by Stryker-compatible tools | ‚úì |

| `killedBy` test name renders in detail panel | ‚úó wrong type |
